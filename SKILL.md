# 用户研究标准作业流程（SOP）V2.0

> **升级说明**：V2.0 在原六阶段线性流程基础上，引入 Agent Loop 多模型交叉验证机制，将SOP从"人工自查"升级为"AI辅助自动纠错"。适用于问卷调研、用户访谈、行为数据分析及竞品研究四类场景。

---

## 核心理念：Agent Loop 质量控制

### 什么是 Agent Loop

Agent Loop（智能体循环）= 让AI像研究员一样**"产出→审查→验证→修正→再产出"**的循环机制。在用研SOP中，每个关键产出节点自动触发多模型交叉验证，确保研究质量不依赖单一视角。

### 三角色协作模型

| 角色 | 职责 | 推荐模型 |
|------|------|---------|
| **主模型** | 框架搭建、初稿产出、结构化整合 | Claude / GPT-4o / mimo-v2.5-pro |
| **审查模型** | 事实核查、逻辑漏洞、遗漏维度、反例检验 | DeepSeek v4-pro / Grok |
| **数据源** | 外部信息补充、实时数据验证 | Tavily API / 天行API / Perplexity |

### Loop 嵌入节点

```
需求承接 ──→ 研究设计 ──→ 数据采集 ──→ 数据分析 ──→ 报告交付 ──→ 收尾
   ↑ [L1]        ↑ [L2]                   ↑ [L3]        ↑ [L4]
   │              │                        │              │
   └──── Loop ────┘────────────────────────┘──────────────┘
         每个 [L] 节点触发：
         ① 主模型产出初稿
         ② 审查模型找错（事实/逻辑/遗漏）
         ③ 外部数据源交叉验证
         ④ 人工确认 → 进入下一阶段
         ⑤ 若审查发现重大问题 → 回到①修正
```

| Loop节点 | 触发时机 | 审查重点 | 审查模型建议 |
|----------|---------|---------|-------------|
| **L1** | 需求确认表填写完成后 | 方法选择是否合理、有没有遗漏研究方法、时间节点是否现实 | DeepSeek（逻辑推理强） |
| **L2** | 问卷/提纲初稿完成后 | 题目措辞歧义、选项覆盖度、跳题逻辑、探针有效性 | Claude（语言敏感度高） |
| **L3** | 初步分析结论产出后 | 关键数字是否有外部来源交叉验证、结论是否过度推断 | DeepSeek + Tavily（事实核查） |
| **L4** | 报告初稿完成后 | 行动建议是否可执行、受众敏感措辞、数据引用准确性 | 双模型交叉审查 |

### Loop 执行规范

1. **审查模型必须给出具体修改建议**，不接受"整体不错，建议优化"这类空话
2. **事实错误必须标注来源**，不接受无出处的纠正
3. **遗漏维度必须说明遗漏理由**，不接受"可以考虑XX"这类模糊建议
4. **人工确认是最终关口**——模型之间的交叉验证解决执行质量，方向正确性由研究员判断
5. **最多3轮循环**——若3轮后仍有分歧，提交人工决策，不陷入无限修改

## 阶段一：需求承接

### 1.1 需求确认三要素

接到研究需求后，填写确认表，三项均明确后方可启动：

| 确认维度 | 确认内容 | 备注 |
|----------|---------|------|
| 研究目的 | 支撑哪个业务决策？决策节点在何时？ | 目的不明确则退回对齐 |
| 交付标准 | 产出物形态（结论页/完整报告/PPT） | 提前确认受众：内部/外部合作方 |
| 时间节点 | 数据回收截止日期 / 报告提交日期 | 倒排关键里程碑 |

**→ [L1 Loop] 需求确认表填写后，提交审查模型验证方法选择合理性**

### 1.2 方法选择判断树

| 研究问题类型 | 推荐方法 | 典型场景 |
|-------------|---------|---------|
| 探索性：不知道用户在想什么 | 定性优先（深度访谈） | 新产品方向探索、痛点挖掘 |
| 验证性：对假设进行量化确认 | 定量为主（问卷） | 用户满意度、功能偏好排序 |
| 竞品/市场缺口 | 桌面研究 + 文本聚类 | 竞品策略分析、口碑对比 |
| 全链路判断 | 定性预实验 → 定量验证 | 大规模投放前的维度聚焦 |

### 1.3 踩坑预警

| 常见错误 | 后果 | 正确做法 |
|---------|------|---------|
| 需求方说"随便看看"就启动 | 做完没人用 | 必须明确支撑什么决策，否则退回 |
| 没确认受众就开写报告 | 给管理层的报告写成技术文档 | 先确认：内部/外部/执行层 |
| 只确认了目的没确认时间 | 做完发现已过决策节点 | 时间节点必须倒排里程碑 |

## 阶段二：研究设计

### 2.1 样本量设计规则

- 各分群有效样本下限：≥500份（重点分析群体建议≥1,000份）
- 投放总量 = 目标有效样本 ÷ 预估回收率（参考基准：问卷回收率3%–5%）
- 两阶段投放：定向投放保底（达目标80%）→ 通投补量，防止小众群体欠采
- 多客群项目：各群独立配额，不以总量替代分群保底

**实战案例：校园游戏调研（N=601）**
- 目标样本600份，实际回收601份，回收率约4%
- 6类人群独立配额，每群≥80份
- K-Means聚类后发现第6类样本量偏小（n=52），分析时单独标注置信度

### 2.2 问卷设计规范

- 结构：通用模块（全员必答）+ 分群跳转模块，通用题占比 ≤30%
- 首题设置身份确认/筛选题，确保分群准确
- 题型优先级：场景描述题 > 强迫单选优先级题 > 量表题
- 强迫选择题避免出现「无偏好」选项，确保数据有区分度
- 完成前检查：跳题逻辑是否完整、必答题是否合理、问卷总时长 ≤8分钟

**→ [L2 Loop] 问卷初稿完成后，提交审查模型检查措辞歧义和选项覆盖度**

### 2.3 踩坑预警

| 常见错误 | 后果 | 正确做法 |
|---------|------|---------|
| 强迫选择题出现"无偏好"选项 | 数据无区分度，分析时全部归入"无偏好" | 去掉该选项，强制用户做出选择 |
| 通用题占比过高（>50%） | 问卷冗长，中途弃答率飙升 | 通用题≤30%，其余按分群跳转 |
| 没有首题筛选 | 非目标用户混入样本 | 首题必须是身份确认/筛选题 |
| 跳题逻辑没测试 | 部分用户看到不该看的题目 | 投放前至少走3遍完整流程 |

### 2.4 访谈提纲设计规范

- 按受访者状态分版设计（如：已购/未购/流失用户不同版本）
- 提纲结构：暖场（2min）→ 行为回顾（8min）→ 核心议题（15min）→ 补充探索（5min）
- 每个核心问题配备1–2个追问探针，避免停留在表层回答
- 不在提纲中预设答案方向，保持开放式问法

**→ [L2 Loop] 提纲初稿完成后，提交审查模型检查探针有效性**

## 阶段三：数据采集

### 3.1 问卷投放

- **投放前**：确认渠道资源位已申请、定向字段已配置、问卷链接已测试
- **投放中**：每日监控回收进度，各分群单独追踪，发现欠采及时补投
- **投放后**：导出数据前检查后台回收总量与分群分布是否符合预期

### 3.2 用户访谈执行

- **访谈前**：提前1天发送确认短信，准备录音设备/在线会议链接
- **记录规范**：实时填写结构化记录表，三栏必填：原话引用 / 行为归因 / 研究者判断
- **有效性标准**：时长≥10分钟且核心模块均已覆盖，否则标注存疑并补充样本
- 每批访谈完成后当日产出阶段小结，不积压至最终分析

### 3.3 行为数据获取

- 提数需求格式：注明字段名称、时间范围、过滤条件、数据用途
- 接收后优先核查口径：日期范围对齐、去重逻辑一致、分母定义统一
- 发现口径偏差立即反馈数据侧，**不使用存疑数据输出结论**

### 3.4 踩坑预警

| 常见错误 | 后果 | 正确做法 |
|---------|------|---------|
| 没有每日追踪分群进度 | 投放结束才发现某群严重欠采 | 每日看板，欠采群及时补投 |
| 访谈记录积压到分析阶段 | 原话遗忘，归因失真 | 当日产出阶段小结 |
| 接收数据没核查口径 | 用了"按会话去重"的数据做"按用户去重"的分析 | 接收后第一件事：核查口径三要素 |

## 阶段四：数据分析

### 4.1 定量分析步骤

| 步骤 | 操作内容 | 质控要点 |
|------|---------|---------|
| 1. 数据清洗 | 剔除无效样本（答题时长异常/关键题全空/逻辑矛盾） | 记录剔除数量和原因 |
| 2. 描述统计 | 各分群基础分布，确认样本代表性 | 检查各群样本量是否达标 |
| 3. 交叉分析 | 分群差异对比，聚焦核心维度 | 避免过度拆分导致样本过小 |
| 4. 漏斗分析 | 适用于行为数据，定位各节点流失量级 | 确认分母口径一致 |
| 5. 数据校验 | 关键数字交叉比对，核查来源差异 | 来源差异>5%须说明原因 |

**→ [L3 Loop] 初步结论产出后，提交审查模型+外部数据源交叉验证关键数字**

### 4.2 定性分析步骤

| 步骤 | 操作内容 |
|------|---------|
| 1. 逐条归因 | 每条访谈提取核心行为动机，不批量归纳 |
| 2. 主题聚类 | 跨样本归并共性发现，标注出现频次 |
| 3. 反例检验 | 主要结论须检验是否存在反例，记录偏差来源 |
| 4. 结论分级 | 核心发现（高频+高影响）/ 次级发现 / 观察备注 |

### 4.3 踩坑预警

| 常见错误 | 后果 | 正确做法 |
|---------|------|---------|
| 清洗数据不记录剔除原因 | 无法复现，被质疑时答不上来 | 每次清洗记录：剔除数量+原因+占比 |
| 交叉分析过度拆分 | 单格样本<30，结论不可信 | 单格<30时合并或标注置信度 |
| 跳过反例检验 | 报告被挑战时措手不及 | 每条核心结论必须找反例 |

## 阶段五：报告产出与交付

### 5.1 报告结构规范

- 固定结构：**核心结论前置 → 分模块展开 → 综合建议收底**
- 结论层次：洞察提炼 → 结论判断 → 行动建议，三层缺一不可
- 禁止描述性堆砌：输出管理结论，不是数据复述
- 受众适配：内部管理层结论前置不铺背景；外部合作方措辞留余地

**→ [L4 Loop] 报告初稿完成后，双模型交叉审查：事实准确性+建议可执行性**

### 5.2 交付前自查清单

| 检查项 | 标准 |
|--------|------|
| 核心结论可见性 | 主结论在第一屏/第一页清晰呈现 |
| 数据来源标注 | 所有引用数据注明来源和时间范围 |
| 关键数字二次校验 | 报告中关键数字与原始数据核对无误 |
| 行动建议完整性 | 每项核心发现均对应可执行建议 |
| 受众敏感内容处理 | 外部报告已对敏感措辞做适当处理 |
| 可交付性判断 | 可直接提交领导/合作方，无需二次加工 |

### 5.3 踩坑预警

| 常见错误 | 后果 | 正确做法 |
|---------|------|---------|
| 结论堆数据不给判断 | 领导看完不知道该做什么 | 每条结论必须有"所以呢"的行动建议 |
| 给管理层铺太多背景 | 前3页还没进入正题 | 管理层报告：第一页就是结论 |
| 关键数字没二次校验 | 报告里的数字和原始数据对不上 | 每个关键数字回溯原始数据核对 |

## 阶段六：项目管理与收尾

### 6.1 进度管理原则

- 启动时识别所有外部依赖节点（数据审核、资源位审批等），提前锁定截止时间
- 外部节点不等待，并行推进其他可进行环节
- 进度阻塞时主动上报：说明卡点原因、影响范围、建议解决路径
- 多线并行项目：每条线单独维护进度状态，不混合汇报

### 6.2 项目收尾

- **落地跟踪**：确认结论是否被产品/运营侧采纳执行
- **文档归档**：问卷原件、访谈记录、清洗后数据集、最终报告均存档备查
- **方法论沉淀**：问卷模板、访谈提纲存入复用库，下次同类项目可直接调用

## 子流程A：竞品研究专项

> 适用于：竞品策略分析、口碑对比、市场缺口研究

### 流程

| 步骤 | 操作 | 质控要点 |
|------|------|---------|
| 1. 选品 | 三层漏斗：直接竞品→间接竞品→标杆 | 不只盯着同行，关注跨品类替代方案 |
| 2. 建框架 | 四种框架按需选：功能对比/体验对比/策略对比/用户口碑 | 根据研究目的选框架，不套模板 |
| 3. 数据采集 | 桌面研究+文本聚类+用户评论采集 | 来源至少3个，单一来源不可信 |
| 4. 找Pattern | 归成四类：机会点/差距点/行业难题/差异化优势 | 每个Pattern必须有数据支撑 |
| 5. 出结论 | 结论先行，每条有数据、路径、风险 | 落地到"我们能做什么" |

**实战案例：小红书宽带竞品分析**
- 竞品：电信天翼云游戏、移动咪咕快游
- 数据源：飞瓜数据+小红书UGC+运营商官网
- 关键发现：竞品100-150GB通用流量 vs 某运营商30GB定向流量，价格带差距30-40元/月
- 落地：输出权益差异化建议，推动流量包结构调整讨论

## 子流程B：漏斗/转化分析专项

> 适用于：行为数据分析、转化链研究、流失定位

### 流程

| 步骤 | 操作 | 质控要点 |
|------|------|---------|
| 1. 定义漏斗 | 明确各节点定义+分母口径 | 节点定义必须业务方确认 |
| 2. 数据提取 | 按节点提取转化/流失数据 | 去重逻辑统一，口径偏差立即反馈 |
| 3. 流失定位 | 找到流失量级最大的节点 | 不只看比例，更要看绝对量 |
| 4. 归因分析 | 对流失最大节点做用户分层 | 区分"不能转化"和"不愿转化" |
| 5. 干预建议 | 针对可干预节点提出优化方案 | 每条建议估算ROI |

**实战案例：宽带转化链分析（211K会话）**
- 漏斗：广告曝光→意向单→企微客服→号卡/宽带转化
- 数据量：月均10万+会话，分4个月清洗
- 关键发现：企微客服→转化环节流失率最高，归因到响应时间和话术质量
- 落地：输出客服话术优化方案+响应时间SLA建议

### 附录E：Agent Loop 审查Checklist模板

**L1 需求审查：**
- [ ] 研究目的是否明确到支撑什么决策
- [ ] 方法选择是否匹配问题类型
- [ ] 时间节点是否现实
- [ ] 有没有遗漏的研究方法

**L2 设计审查：**
- [ ] 问卷题目措辞是否无歧义
- [ ] 选项覆盖度是否完整
- [ ] 跳题逻辑是否经过测试
- [ ] 访谈探针是否能挖到深层动机

**L3 分析审查：**
- [ ] 关键数字是否有外部来源交叉验证
- [ ] 结论是否过度推断（数据→洞察→结论→建议，每步都要有依据）
- [ ] 是否存在反例未被记录
- [ ] 分群样本量是否达标

**L4 报告审查：**
- [ ] 结论是否在第一页可见
- [ ] 每条建议是否可执行
- [ ] 数据引用是否标注来源
- [ ] 受众敏感措辞是否已处理
