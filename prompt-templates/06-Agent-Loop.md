# 结构化对抗审查

## 目的

使用独立上下文和明确检查标准，对研究材料、分析和报告进行反例检查。

结构化对抗审查可以发现：

- 逻辑遗漏；
- 过度推断；
- 方法不匹配；
- 证据不足；
- 反例缺失；
- 建议不可执行。

它不等于独立事实验证。

只有引入新的数据、原始材料、可靠来源、重新计算或人工核查，才能提高事实证据强度。

## 基本流程

```text
初稿
→ 作者角色自查
→ 独立审查角色逐项找问题
→ 对审查意见进行证据核验
→ 接受、拒绝或修改
→ 最终质量门
````

## 审查角色

审查者不得只评价“整体不错”。

必须逐项输出：

* ✅ 通过；
* ⚠️ 建议修改；
* ❌ 阻断问题；
* 🔎 需要补充证据。

## 通用审查提示词

```text
请以独立审查者身份审查以下内容。

待审查内容：
[粘贴内容]

项目决策：
[说明该内容需要支持什么决策]

请检查：

1. 是否真正回答决策问题；
2. 方法是否能回答研究问题；
3. 是否混淆事实、推论、待验证和建议；
4. 是否存在无证据结论；
5. 是否把相关性写成因果；
6. 是否用小样本推广到总体；
7. 是否遗漏关键反例；
8. 是否存在数据口径或样本问题；
9. 建议是否有明确动作、指标和前置条件；
10. 是否存在伦理、隐私或敏感信息风险。

逐项输出：
- 状态：✅ / ⚠️ / ❌ / 🔎
- 问题：
- 原因：
- 具体修改：
- 需要补充的证据：

最后给出：
- 是否通过；
- 阻断问题数量；
- 建议修改数量；
- 尚未验证的事实数量。
```

## 审查意见处理

每条审查意见标记：

* 接受；
* 部分接受；
* 拒绝；
* 待补充证据。

不得因为审查来自另一个模型就自动接受。

## 停止规则

最多执行三轮。

三轮后仍存在以下问题时，提交人工决策：

* 方法选择分歧；
* 事实来源冲突；
* 业务取舍冲突；
* 伦理风险；
* 证据无法获得。
